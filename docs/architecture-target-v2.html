Discovery Engine — wie die neue Version arbeitet

Ein System, das Stellenanzeigen aus vielen Quellen einsammelt, aufräumt und jeder Person die passenden zeigt.

🏭 Bild dazu: Stell dir ein Lager vor. Zuerst wird alles reingetragen, was reinkommt (nichts wird vorab weggeworfen). Dann wird einmal für alle aufgeräumt — Doppeltes raus, fehlende Infos ergänzt, ordentlich ins Regal. Erst ganz am Ende bekommt jede Person ihre eigene Auswahl aus dem sauberen Regal. Die drei Schritte laufen getrennt — wenn eine Quelle hakt, läuft der Rest weiter.

Die wichtigen Weichenstellungen

Jeder ? ist eine bewusste Entscheidung darüber, wie mit den Daten umgegangen wird — kein technisches Klein-Klein. offen = noch nicht entschieden, sollte vor dem Bauen geklärt werden.

1 Erst alles aufheben, dann aussortieren

Früher wurde sofort weggeworfen, was nicht passte — weg war weg. Jetzt landet erst alles im Rohlager.

+ Vorteil: nichts geht verloren. Man kann später neu bewerten, nachzählen, Fehler nachvollziehen.

2 Anzeige am Stück aufheben statt sofort zerlegen offen

Jede Anzeige hat 9 einheitliche Felder (Titel, Link, Firma, Ort, Beschreibung, Gehalt …) + die komplette Original-Antwort der Quelle. Erst beim Aufräumen wird in Spalten zerlegt.

+ Vorteil: braucht später jemand ein Feld, das wir früher ignoriert haben — es steht noch im Original.

Original sieht je Quelle anders aus:
• Firmen-Portale (Greenhouse & Co.) → reich: voller Beschreibungstext, Abteilung, stabile ID.
• Sammel-Portale (Adzuna & Co.) → flacher: Weiterleitungs-Link, oft nur Kurztext, mit Gehalt.

⚠ Heute inkonsistent: Firmen-Portale legen das Original ab, Adzuna lässt es leer. → beim Umbau: jede Quelle muss ihr volles Original ablegen, sonst trägt das „alles aufheben" nicht.

3 Wo merkt sich das System bekannte Jobs? entschieden: A + D

Heute liegt das Gedächtnis nur im Arbeitsspeicher → nach Neustart vergessen, parallele Läufe blind füreinander.

→ A: Gedächtnis wandert in den dauerhaften Speicher (Datenbank). D: nur die letzten Wochen merken (alte Jobs laufen eh ab). Der alte Spezial-Code wird überflüssig. Details in der Tabelle unten.

4 Altes Regal umbauen oder neues danebenstellen? entschieden: Weg B

Das heutige Regal ist für eine Person gebaut und nur zum Dazulegen. Das neue muss „für alle gleich" + Lebenslauf je Job können.

→ Weg B: neues Regal daneben bauen, altes läuft parallel weiter, Daten rüberkopieren, vergleichen, dann automatisch per Stichtag umschalten, erst dann altes löschen.

+ Jeder Schritt einzeln rückrollbar, App hakt nie. Langsamer — aber Sicherheit zählt, Tempo nicht.

5 „Noch offen oder schon weg?" merken

Ein Job ist ein Zustand, kein einmaliges Ereignis. Das System merkt sich, wann es ihn zuletzt gesehen hat.

+ Vorteil: erkennt automatisch, wenn eine Stelle nicht mehr ausgeschrieben ist (früher unmöglich).

6 Pro-Person-Auswahl jetzt · eigener Zeitplan später offen

Jetzt bauen: die getrennte Trefferliste pro Person — und den Ablauf, der über alle Personen läuft.

Später: jeder Person einen eigenen Zeitplan geben — lohnt erst bei vielen Nutzern.

+ Vorteil: neue Person funktioniert sofort, nur die Feinsteuerung wächst später mit.

7 Suchwörter gehören einer Person — und sind löschbar entschieden

Einige Quellen (Adzuna, Jooble, Indeed, LinkedIn) muss man mit Suchwörtern fragen — das passiert schon beim Einsammeln. Ein Job, nach dem nie gesucht wurde, liegt gar nicht erst im Lager.

→ So gelöst: Jedes Suchwort gehört einer Person (eigene kleine Liste search_terms). Eingesammelt wird mit allen Suchwörtern aller Personen zusammen. Nutzen zwei dasselbe Wort → einmal gefragt, zwei Besitzer.

+ Person löscht ihr Wort / wird gelöscht → ihr Eintrag weg → Wort wird nur weiter geholt, wenn eine andere Person es noch hat, sonst fällt es automatisch raus. Sauber zuordenbar + löschbar (DSGVO).

Nur tatsächliche Nutzer ergänzen Suchwörter. Ein paar Tage Verzögerung bis ein neues Wort greift = ok (Firmen-Portale zeigen ohnehin sofort eine breite Auswahl).

Weichenstellung ③ genauer — das Gedächtnis für „Fast-Doppel"

Worum es geht: Derselbe Job taucht oft über mehrere Portale auf, mit leicht anderem Text. Um das zu erkennen, braucht das System ein Gedächtnis aller schon gesehenen Jobs. Die vier Optionen unterscheiden sich nur darin: wo dieses Gedächtnis liegt — und davon hängt ab, ob es einen Neustart übersteht.

OptionWo liegt das Gedächtnisübersteht Neustart?mehrere Läufe gleichzeitig sicher?macht alten komplizierten Teil überflüssig?Haken
A · in der Datenbank
(empfohlen)		 direkt im dauerhaften Speicher (Datenbank-Suche nach ähnlichem Text) jajaja ✂ findet Ähnlichkeit etwas anders → einmal nachjustieren; bei sehr langen Texten etwas langsamer
B · eigenes Gedächtnis-Programm ein extra Dienst nur fürs Merken jajanein ein zusätzliches Teil, das laufen und gewartet werden muss
C · jedes Mal frisch aufbauen kurz im Arbeitsspeicher, vor jedem Lauf neu aus dem Regal geladen entfällt (immer frisch)janein je größer das Regal, desto langsamer der Start jedes Laufs
D · nur die letzten Wochen merken
(Verstärker)		 egal wo — nur ein kurzer Zeitraum +++ keine Lösung für sich; mit A oder C kombiniert hält es das Gedächtnis klein (alte Jobs laufen eh ab)

Empfehlung A + D: Das Gedächtnis wandert vom flüchtigen Arbeitsspeicher in den dauerhaften Speicher — genau passend dazu, dass jetzt sowieso „erst alles aufheben" gilt. Der alte, komplizierte Spezial-Code wird überflüssig und kann weg. Exakte Doppel fängt die Datenbank ohnehin selbst ab. Braucht man später doch mehr Genauigkeit → Variante C + D (statt des extra Dienstes B).

Der Kern-Unterschied in einem Satz

Früher: alles wurde sofort durchgefiltert, nur die „Gewinner" blieben übrig — der Rest war für immer weg, und das galt nur für eine Person.
Jetzt: erst alles aufheben → einmal für alle aufräumen → für jede Person passend zuordnen. Genau die Trennung „für alle gleich" gegen „pro Person" ist der Schlüssel, der Mehr-Nutzer überhaupt möglich macht.
